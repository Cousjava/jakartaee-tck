1.0 Introduction

    1.1 Overview

This test plan documents the deliverables and milestones for the test development relevant to the Enterprise JavaBeans 2.1  specification which is a requirement of the J2EE 1.4 platform specification. This includes the EJB tests developed for this specification and other deliverables specific to each quality groups' infrastructure to ensure compliance, robustness, integrity, and overall quality and consistent look and feel to the certification and reference implementation quality processes. The compatibility test cases are derived from the requirements of the engineering specifications. The quality test cases and test applications are derived from optional behavior and implementation specific functionality but still based on specification adherence and reference implementation design plans.

    1.2 Objectives

The objectives of this plan is to highlight the collaborative test effort for the Enterprise JavaBeans 2.1 component of the J2SDKEE 1.4 reference implementation to ensure it meets its' quality and conformance goals.  The reference implementation will validate that the specifications can be implemented which, in turn, provides a reference point for our test development community.   The test data generated from this effort will provide an overall quality assessment for release readiness for J2SDKEE 1.4 and the effectiveness of the Compatibility Test Suite on any implementation of the Enterprise JavaBeans 2.1 specification.

2.0 Features to be tested by CTS

Provide compatibility tests for the set of new functionality in the Enterprise JavaBeans Specification Version 2.1  provided as an extension to the existing Enterprise JavaBeans Specification Version 2.0. Also additional test coverage which was not covered in the previous release due to time and schedule constraints will also be highlighted. This functionality includes:
  • Select set of existing CTS tests built on the 1.3.1 platform for testing application portability.
  • New tests for EJB Timer Service
    • Stateless Session and Entity Beans
    • CMT and BMT beans
    • ejbTimeout()
    • Single-event and interval timers
    • Duration and date timers
    • Timer handles
    • Operations allowed for Timer Service methods
  • New tests for EJB 2.1  EJB QL functionality
    • Null Semantics
      • Null values returned from finder and select methods
      • "Inner join" semantics for handling null cmr-field values
    • ORDER BY clause
    • Aggregate functions
    • Functional expression:  MOD
    • Additional return value types:  cmp-field or result of aggregate functions
    • Input Parameters for  IS [NOT] NULL expressions
    • Input Parameters and numeric literals for IN expressions
  • EJB Security
    • SSL3.0 & TLS 1.0 Cipher suites
  • Secure Interoperability (CSIv2) Tests
    • validate target name in Client authentication token
    • check multiple policy support
    • log changes (target requires=<number> to string)
    • login (keystore, password, privatekey)
  • Web Service Endpoint Interface
    • Integrate JSR 109 TCK EJB tests into CTS 1.4
    • Identify coverage deltas and provide additional tests as needed
    • Web Service Endpoint Component Interface Allowed Methods Tests (stateless session beans)
    • SessionContext.getMessageContext()
  • Enhanced EJB Interoperability Scenarios
    • Transaction interoperability with CMT test driver beans
    • End-to-end round trip interoperability scenarios
    • CORBA view of an EJB
  • New Deployment Descriptors
    • Message-Driven Bean's Configuration Properties (activation-config requirements)
    • Web Service references and service-ref deployment descriptor element
  • Sweep tests using CMP beans with deprecated TX attributes of attributes of Supports, Never, NotSupported ( Full Problem Analysis)

7.0 Test Methodology

    7.1 CTS

The EJB compatiblity tests consist mainly of end-to-end test scenarios focusing on the container's ability to handle specified requirements.  The enterprise archive is deployed to the EJB server and then test methods are called from the Client.    The beans are coded to specification but the beans conformance is not explicitly tested but ensured by verifying each archive prior to external milestone releases using the reference implementation's application verifier.  Additionally, the CTS will perform an additional validation step to ensure the deployment descriptors conform to the specified DTD (either 1.2, 1.3, or 1.4).   CTS will also provide  EJB backward-compatibility tests that comprise a subset of tests that are built on the previous version of the reference implementation to ensure licensee implementations still support this portable archive.  There are 3 types of tests that make up the compatibility test suite.  These include:
  • Signature Tests - Checks that all of the required API signatures are available for an implementation, no supersets or subsets.
  • API Tests - Verifies that all of the required classes and methods behave according to the specification on a method by method basis.
  • End to End Tests - A type of compatibility test which was designed specifically for the J2EE platform. These type of tests span across n-tiers of the J2EE platform.
The Signature tests will need to be regenerated for the API changes and additions in EJB 2.1 that were not in EJB 2.0.  These need to be generated and tested prior to each external milestone release.

There are limitedAPI-type tests for EJB2.1.  The API's are tested implicitly via the end-to-end.   However, assurance of coverage metrics with respect to the EJB APIs will be assessed. If there are any APIs that are not specifically covered in the end-to-end tests, they will be added.

The test teams will use nightly and promoted releases of J2EE 1.4 reference implementation to develop EJB  tests.

For CTS, the generation of runtime XML and deployment of the test .ears will be driven by the JavaTest 3.0 harness. JavaTest 3.0 will also manage the logging and reporting of test results.  It is a requirement, as defined by the J2EE rules, that the licensees use the JavaTest harness for certification.
 

7.1.1 CTS Infrastructure and Quality Goals

Additional work will be required in the 1.4 development time frame to enhance the quality and maintain the CTS build infrastructure and overall architecture.  These tasks are:
 
  • Enhance EJB test design to include wrapper classes that will facilitate test development where the same test logic is shared by different components.
  • For EJB tests requiring CMP beans, only CMP 2.x beans will be used.  We will maintain existing CMP 1.1. beans in the test suite as necessary, ensuring they are compliant to the 2.1 spec.
  • Perform assertions reviews for existing and new tests (see Section 8.0).
  • Adjust the DBSupport utility class to be used by all the tests.
  • Hold design reviews for new features as needed followed by code review of the implementation. See  the Design and Code Review Process document for the team guidelines.
  • Ensure all .ears pass the J2EE RI verifier prior to any milestone release
  • Ensure setup and cleanup methods within each test are used properly and enhance test execution by ensuring the dependencies or resources for the tests are available prior to test execution.
  • Assist licensees by commenting code better to trace test logic.
  • Change the build infrastructure of the EJB tests to ANT consistent with the architecture defined for the entire CTS.
  • Hand edit XML deployment descriptors for EJB test packages.

    • 7.1.2 Dependencies

    The following dependencies are necessary for the development of these tests:
    • Well defined determination of the XML strategy for J2EE 1.4
    • J2EE Reference Implementation for validation of tests
    • XML editing tools to support editing of  XML Descriptions

    7.1.3 Documentation

    The documents used in the development process including XML files containing the specification assertions and the JavaDoc APIs will be located in the CTS workspace under $TS_HOME/internal/docs/ejb.  The test plan (this document) will also be in this location to keep the document revisions in a controlled environment. This allows multiple users to access the document with historical change information captured.

    7.1.4 Targeted Coverage

    The CTS will target 50% test coverage for the specification and 70% test coverage for JavaDoc APIs in its' deliverable.
    90% of the CTS tests must pass the reference implementation for a GO vote on an external milestone release.

    7.2 SQE

    7.2.1 Approach

    An application centric approach will be used to test all the features listed in the "3.0 Features to be tested by SQE" section..   The test design documents utilizes use-case  methodology, static class diagrams and implementation diagrams from UML to describe the test design. The document templates and other relevant formats are borrowed from the templates designed for the group.

    An integrated application called SuperBank has been designed to cover major part of the J2EE1.4 testing. For detailed design document, see SuperBank Test Design Specification . New application designed for testing J2EE1.4 Platform specification will target bank centric business logic and will be part of the integrated SuperBank application.  For the features where the tests cannot be integrated into the SuperBank application, independent and focussed test applications will be developed to test those features.

    The test team will use nightly and promoted releases of J2EE1.4 reference implementation to develop J2EE1.4 tests. The new code repository will be CVS based and the build environment will be based on XML.

    For generic messaging for MDBs, the objective is to have an EIS send inbound messages to the J2EE server which would be received by a JCA 1.5 compliant JMS adapter. The adapter in turn dispatches the messages to MDB. This model should work for any messaging system as long as an appropriate 1.5 compliant Resource adapter is available to support dispatch to the MDB. Testing related to this feature of MDB is convered in the quality and design plan of the JCA 1.5 (Java Connector Architecture). Please refer to these docs for detailed info. They can be accessed at the web site below.
    http://j2ee.east/sqe

    For EJB QL features and webservices support, the objective is to test the RI w.r.t EJB QL and webservice endpoint model. The integration application would be aimed at testing the real world usage of these components.

    For EJB Timers, the main objective is to fit the features in an application environment whereby the timers will be used to perform certain tasks intrinsic to the application design in real world. In the process, an attempt will be made to find out any problems with the Timers used in an integrated fashion with other types of components in the application.
     

    7.2.2 Goals

    •  To make sure that the reference implementation is a complete (see features in Section "Features to be Tested") and  working system and the underlying technology is fairly robust.
    •  To analyze the implementation versus the specification, identify potential inconsistencies, vagueness and bugs. Communicate the same to the architects and developers through bug reports.


    7.2.3 Test frequency and Categories

    •     Nightly tests: Automated test suites run nightly on supported platforms
    •     Weekly promoted tests: It includes nightly runs and subset of manual tests
    •     Full test cycle: All automated tests, manual tests and installation tests.

    8.0 Assertion Generation and Reviews

    Each EJB test is mapped to a specific assertion as defined in the EJB 2.1 specification.  Specification assertions will be generated starting with the distribution of the expert group drafts.  With each distribution, the affected areas will be updated to assure the testable and optional assertions in the specification are highlighted.  The assertions will be generated using XML tools and transformed to HTML as the output document.  This document will be reviewed with the specification lead, test team members, and the reference implementation leads to ensure the validity of the identified test targets.  Once reviewed, the tests will be tagged with the appropriate unique assertion id (one or more).  This allows coverage analysis to be performed to determine overall test coverage that compatibility and quality tests provide against the EJB specification.  Combined with all component coverage data, this will allow representation of overall test coverage the platform can achieve.  Based on analysis data, it is possible additional tests will need to be written to ensure the required component coverage for both specification and JavaDoc assertions. For a list of the assertions identified in the specification, see  EJB 2.1 Specification Assertions .  For a list of testable JavaDoc APIs for EJB 2.1, see  EJB 2.1 JavaDoc Assertions .  For a view of the coverage metric data derived from the review meetings and test tagging, see  EJB 2.1 Specification Coverage Summary  and  EJB 2.1 API Coverage Summary , respectively.

    11.0 Assumptions/Risks

    • Coverage metric data may not yield required CTS coverage requirements. This is considered to be low risk as EJB has been the cornerstone of the platform and the test development focus for the last two releases.
    • WebServices for J2EE TCK may not contain expected coverage.  This is higher risk than anticipated as the deliveries are not on schedule.  Assessing impact weekly.  Weekly meetings are now being held to keep project on track.  The above test development tasks for EJB web services were derived from these meetings.
    • New backend databases to test with. This is a low risk for EJB QL tests.
    • Some technology specs are still in early stages and significant spec changes may be expected in future drafts.  Any changes made late in the release cycle may cause instability and low coverage in that area by SQE.
    • Timely delivery of stable weekly promoted builds is very important for SQE.
