1.0 Introduction

    1.1 Overview

This test plan documents the strategy and deliverables for the compatibility test development effort for the Enterprise JavaBeans (EJB) specifications which are a requirement of the JavaEE 5.0 platform specification.  The EJB requirements will be defined in three documents:
  • Enterprise JavaBeans Core Contracts and Requirements
  • EJB 3.0 Simplified API
  • Persistence API
Collectively, these specifications redefine the EJB architecture by simplifying the APIs defined in earlier versions of the specifications emphasizing an ease-of-development model consistent with the platform for enterprise application developers.  The compatibility tests written to verify the requirements defined in the above specifications will be delivered as part of JavaEE 5.0 Compatibility Test Suite (CTS).  In addition, a subset of these compatibility tests for the Persistence API specification will be delivered as a separate standalone Technology Compatibility Kit (TCK) to certify the requirements of the use of such APIs in J2SE (non-JavaEE) environments.  The compatibility test cases are derived from the requirements of the technology specifications.  Optional behavior and implementation specific functionality will be identified for inclusion in our product quality test suite via the assertion review process (See Sections 3.0 and 7.0).

    1.2 Objectives

The objectives of this plan is to define a test strategy for the Enterprise JavaBeans and Persistence components of the JavaEE 5.0 reference implementation to ensure they meet their conformance goals.   It is crucial that the compatibility tests themselves are designed and delivered as portable applications so this will be considered while defining the test architecture as well as during test development.  The CTS EJB test development team will strive to achieve high quality releases for both of the targeted deliverables. The test development tasks will be scheduled to align with reference implementation feature integration schedules to 1) have a platform to exercise the tests against as they become ready for execution, and 2) measure compatibility and quality throughout the reference implementation software development cycle.  The test reports generated from this effort will provide an overall compatibility and release readiness assessment for these requirements of the JavaEE 5.0 platform and Persistence reference implementations.

  • A set of compatibility tests for Enterprise JavaBeans Core Contracts and Requirements deltas from EJB 2.1,  new tests for the simplified and persistence API specifications
  • Updated sections of the CTS 5.0 User's Guide, as needed

5.0 Test Configurations

The CTS development team will develop and test against  nightly and promoted releases of the JavaEE 5.0 reference implementation and the database shipped with the reference implementation.  Other database configurations will be tested by the CTS QA team as identified in their test execution plan for JavaEE 5.0. 

The test configurations for the standalone Persistence TCK are unknown at this time.   

6.0 Test Methodology

There are several types of tests that comprise the compatibility test suite.  These include:
  • Signature Tests
    • Checks that all of the required API signatures are available for an implementation, no supersets or subsets.  These signatures will be rerecorded for the JavaEE 5.0 platform and recorded for inclusion in the Persistence TCK.
  • API Tests
    • Verifies that all of the required classes and methods behave according to the specification on a method by method basis. The APIs are tested implicitly via the end-to-end tests.   However, assurance of coverage metrics with respect to the EJB APIs will be assessed.  If there are any APIs that are not specifically covered in the end-to-end tests, API tests will be written and added to the test suite.
  • End to End Tests
    • A type of compatibility test which was designed specifically for the JavaEE platform. These type of tests span across n-tiers of the JavaEE platform.
  • Integration Tests
    • End-to-end tests that demonstrate the interaction of a variety of JavaEE components in a test scenario.   These tests will be extended to the EJB 3.0 portion of CTS to be sure various client types interact with 3.0 beans successfully as defined in the Simplified API specification,  Chapter 9 - Compatibility and Migration.  An example of such test scenario is:
      • Client --> JSP --> EJB3.0 --> EJB2.1 --> Database
  • Compat Tests
    • CTS will also provide EJB backward-compatibility tests that comprise a subset of tests that are built on the previous version of the reference implementation to ensure licensee implementations still consume this portable archive.
    • Additionally, the CTS will provide archives that contain EJBs using mixed deployment descriptors that conform to DTD's for 1.2(?), 1.3, 1.4 and 5.0 platforms. This tests the ability to deploy such an archive and validate the behavior of the business methods during runtime.  
Historically, the EJB compatibility tests consist mainly of end-to-end test scenarios focusing on the container's ability to handle specified requirements.  To that end, emphasis on conformance to deployment requirements are met so the enterprise archive can be deployed to any EJB server.   In other words, intentionally defining CTS archives with adverse deployment descriptor information is not a typical task for this effort rather is done as part of the platform specification assertions and the deployment test development efforts.  The beans and entities are coded to specification but the bean or entity conformance is not explicitly tested but ensured by verifying each archive prior to external milestone releases using the reference implementation's application verifier.

6.1.7  Persistence TCK  Design Tasks

The overall architecture of the CTS persistence tests is to be able to execute the tests whether invoked from a component within the JavaEE environment or from a standalone client in J2SE environment.  These tests, therefore, will be written as "service tests".  This historical term is used to test APIs which are required to be available within different JavaEE containers (i.e., ejb, web).  These APIs could be thought of as "services" available to each container, thus the name "service tests".   New vehicles, namely; stateless3 and stateful3, have been created for the purpose of accessing the EntityManager (via both injection and JNDI lookup) and verifying the specified requirements from these JavaEE components with respect to packaging, deployment, and transactional behaviors.   Because these combinations are extensive and exhaustive,  obtaining the EntityManager and demarcating tranaction boundaries has been abstracted so test execution can occur regardless of environment.  This entails injection of the EntityManager or EntityManagerFactory and delegating the transaction management to the appropriate calls whether container-managed or application-managed.  This is intended to provide coverage for the majority of test cases but certainly more deliberate end-to-ends tests will be identified to exercise the JavaEE container requirements of transaction propagation and extended persistence contexts.

For the EJB QL tests, the CTS 1.4 EJB QL data model has been ported to EJB 3.0 entities and extended to include additional relationships, inheritance hierarchy, and embeddable classes to test the new language extensions.  This design allows us to maintain current query test coverage and extensions in both the JavaEE and standalone implementations.

For other test areas, a subset of these relationships will be used to alleviate complexity of the data model where the assertions don't require such.

Verification to ensure conformance of required APIs for third-party persistence provider needs to be defined.

  • Create persistence workspace from CTS parent workspace and define BOM and build rules for standalone TCK bundling process.
    • release/tools/persistence.xml
    • install/persistence/bin
    • install/persistence/lib (maybe)
  • Identify, create, and maintain assertion documents as specifications are revised.
  • Review the test development tasks with product team to be sure priorities are in line with expectations.
  • Hold peer design reviews for new features as needed followed by code review of the implementation. See  the Design and Code Review Process document for the team guidelines.
  • During test development, ensure that the bean provider's requirements are met per bean flavor and, for each external milestone release, ensure all archives pass the JavaEE AVK. 
  • Ensure setup and cleanup methods within each test are used properly and enhance test execution by ensuring the dependencies or resources for the tests are available prior to test execution.
  • Assist licensees by commenting code better to trace test logic.
  • Review packaging tasks and ensure they meet the requirements of JavaEE 5.0
  • Identify all possible clients and ensure they interact with EJB 3.0 beans (see Section 6.1.6)
  • For CTS, test execution will be driven by the JavaTest 3.0 harness. JavaTest 3.0 will also manage the logging and reporting of test results.  It is a requirement, as defined by the JavaEE rules, that the licensees use the JavaTest harness for certification.

6.3 Documentation

The documents used in the development process including XML files containing the specification and JavaDoc assertions will be located in the CTS workspace under $TS_HOME/internal/docs/ejb3.  The test plan (this document) will also be in this location to keep the document revisions in a controlled environment. This allows multiple users to access the document with historical change information captured. The documents supporting this test development effort will be posted on the EJB Compatibility Test Development Home Page (see Section 9.0 - References).

6.4 Targeted Coverage

The EJB 3.0 and Persistence test coverage target will be 70% for the specifications and 90% for APIs for their initial release.

7.0 Assertion Generation and Reviews

Each EJB compatibility test case is mapped to a required testable assertion as defined in the EJB specifications.  Specification assertions will be generated starting with the distribution of the expert group drafts.  This is a manual task done by reviewing the specification language, highlighting the assertion, then capturing the information in XML format using a pre-defined dtd.   With each revision of the specifications, the assertions need to be reviewed and updated to assure the language matches that of the specification.  These assertion lists are distributed to the licensee community to associate the test case unique assertion id to the specification.  It is intended that the final assertion lists will be reviewed with the specification lead, reference implementation leads, and test team members to assess the validity and proper cataloging of the identified test assertions.  Once the assertion documents are prepared and test cases tagged, coverage metrics can be obtained to assess the test coverage against each specification. 
For an example of the coverage metrics that can be obtained, click on the following links to view the Enterprise JavaBeans 2.1 specification and API summaries.
  • EJB 2.1 Specification Coverage Summary
  • EJB 2.1 API Coverage Summary
The CTS EJB tests will additionally be used in assessing test coverage when the suite is executed against SJSAS product release candidates.

For the EJB and Peristence specifications, there will be one assertion document per specification.   The Core Contracts and Requirements specification will start with the EJB 2.1 assertion document source and will be updated to the current specification language.   The EJB 2.1 assertion documents will remain intact.  For the Simplified and Persistence API specifications, new assertions documents will be generated.  It is intended that the SJSAS AVK team will share the same assertion sources and tag their verifier tests accordingly.  Coordination of this effort is on-going with the AVK team.    For detailed assertion information of the EJB 3.0 specifications to date, follow the links below.
  • EJB 3.0 Simplified API Specification Assertions
  • EJB3.0 API JavaDoc Assertions
  • Enterprise JavaBeans Core Contracts and Requirements (EJB 2.1 Final)
  • Persistence API Specification Assertions
  • Persistence API JavaDoc Assertions

8.0 Release Criteria

For compatibility tests, 90% of the CTS tests must pass the reference implementations for a GO vote for an external beta release. 100% of the CTS tests must pass the reference implementations for an FCS release.  For a beta release, no P1 or P2 bugs can exist unless waived and for FCS, no P1, P2 or P3.

10.0 Assumptions/Risks

  • There are two CTS test development engineers allocated to the project which is assumed to be a risk as the EJB requirements will be delivered in three specification documents.  Coordination of CTS tests are being discussed with SQE EJB team.    Additional requirements have been added since EDR2 specification distributions.  The Public Drafts are currently under review as to impact.  Risk to implementation and delivery will be escalated as needed to the weekly EJB Pteam as public draft requirements are fully evaluated.
