Introduction

This plan documents the high level Compatibility Test Plan for the Java API for XML-based RPC (JAX-RPC) Version 1.0 Specification which defines APIs and conventions for supporting XML based RPC protocols in the Java Platform. Both the Technology Compatibility Kit (TCK) and the Compatibility Test Suite (TS) comprise a suite of tests in which the test cases are derived from the requirements of the JAX-RPC specification. These TCK and TS tests are written to ensure application portability and compatibility among JAX-RPC implementations based on this API. The reference implementation will validate that the specifications can be implemented, this will provide a reference point for the development community, and will be used for developing Compatibility Tests, which leads to a brand. This ensures Sun's, specifically Java's WORA (Write Once Run Anywhere) mission.

Goals

Provide compatibility tests (the Technology Compatibility Kit - TCK) and (the Compatibility Test Suite - TS) for the set of functionality as defined in the JAX-RPC Specification Version 1.0. These tests will verify the JAX-RPC server-side and client-side API's and the JAX-RPC runtime. Marshalling/UnMarshalling of all supported XML data types by JAX-RPC going from wsdl-to-java as well as all supported JAXRPC data types going from java-to-wsdl will be tested and verified. Static and Dynamic JAX-RPC invocations will be tested on the client-side using both a generated stubs based model, Dynamic Invocation Interface API's for invoking a remote procedure call and Dynamic Proxies. Importing and Exporting of WSDL abstract service definitions will be tested for JAX-RPC implicitly via the development of client and server test programs. JAX-RPC testing requires a server side JAX-RPC 1.0 compatible implementation to be either a Web Server Platform which supports the Servlet 2.3 or higher version or a J2EE 1.3 Platform or higher version which supports the Servlet 2.3 or higher and the EJB container.

Non-Goals

  •     Performance
  •     Scalability
  •     Ease of use
  •     Non-required (implementation specific) areas of the specification
  •     No protocol interoperability
  •     Interoperability other than J2EE to J2EE


Types of Compatibility Tests

There are 4 types of tests that make up the Compatibility Test Suite, these include:  
  • Signature Tests - Checks that all of the required API signatures are available for an implementation, no supersets or subsets

  • API Tests - Verifies that all of the required classes and methods behave according to the specification on a method by method basis.

  • End-To-End Tests - A new type of compatibility test that was introduced with the J2EE platform. These type of tests span across N-tiers of the J2EE platform. These tests primarily focus and depend on server-side container runtime environments for proper execution, testing and validation.

  • Interoperability Tests - A new type of compatibility test that was introduced with the J2EE platform. These type of tests span across different J2EE platforms from different vendor's to test interoperability among these J2EE vendor's platforms.

    For the Technology Compatibility Kit (TCK), we will be doing Signature, API, End-To-End tests for JAX-RPC 1.0 technology release.

    For the Compatibility Test Suite (TS), we will be doing Signature, API, End-To-End, and Interoperability tests for the JAX-RPC 1.0 technology release integrated within the J2EE 1.4 platform.



Compatibility Test Suite Releases

There will be 2 types of Compatibility Test Suite releases for JAX-RPC, they are:  
  • JAX-RPC 1.0 TCK - Technology Compatibility Kit just for JAX-RPC 1.0 technology release

  • J2EE 1.4 TS - Compatability Test Suite for J2EE 1.4 Platform which will contain the tests for the JAX-RPC 1.0 technology integrated within the J2EE 1.4 Platform.

    For the JAX-RPC 1.0 TCK we will be using the Apache TomCat 4.0 Web Server Platform for testing or the J2EE 1.3 Platform using its Web Container. Signature, API and some end-to-end testing will be the focus of the JAX-RPC 1.0 TCK.

    For the J2EE 1.4 TS we will be using the J2EE 1.4 Platform for testing. End-To-End testing and Interoperability testing will be the main focus of the J2EE 1.4 TS. The Signature and API tests from the JAX-RPC 1.0 TCK will also be included in the J2EE 1.4 TS.

High Level Test Areas

The following features of JAX-RPC will be tested as specified:

Features Tested for TCK in JAX-RPC 1.0

  • Servlet 2.3 Container Focus for Compatibility Tests
  • APIs of public classes, methods and fields (See detail below)
  • Client/Server Programming model
  • Client Invocation Models (Stub, DII, Dynamic Proxy)
  • Marshall/UnMarshall of XML supported data types (wsdl-to-java)
  • Marshall/UnMarshall of JAXRPC supported data types (java-to-wsdl)
  • Encoded/Literal Data Types
  • Parameter Passing Modes (IN, OUT, IN/OUT)
  • Interaction Modes (Synchronous Request/Response and One-Way)
  • RPC Service with Multiple Service Definition Interfaces
  • RPC Service with Inherited Service Definition Interfaces
  • XML-to-JAVA Name Mapping Tests
  • Import/Export of WSDL
  • Java to XML/WSDL Mapping (Implicit - via import/export testing)
  • WSDL/XML to Java Mapping (Implicit - via import/export testing)
  • Basic Authentication
  • Session Management
  • Application Portability of Client/Service code (Java-to-Java)
  • SOAP Messages With Attachments (Mapping between MIME types and Java Types)
  • SOAP Message Handlers

Features Tested for TS in J2EE 1.4

  • All of the above
  • J2EE Container(s) Focus (EJB/WEB) for Compatibility Tests (JSR 109/J2EE 1.4)
  • J2EE Component Model for JAX-RPC services (JSR 109/J2EE 1.4)
  • Transaction and Security Management if defined (JSR 109/J2EE 1.4)
  • J2EE to J2EE Interoperability (JSR 109/J2EE 1.4)

Features Not Tested

  • No interoperability testing to other vendors products
  • No over-the-wire protocol testing (implicit via end-to-end testing)
  • No tool testing (java-to-wsdl tool or wsdl-to-java tool)
  • No explicit testing of generated code or XML. All to be implicitly tested.
  • No J2SE based JAX-RPC runtime system testing
  • No Pluggable Serializer/DeSerializers (Implementation Specific)
  • No security testing using HTTP/S (Implementation Specific)
  • Service Context (Explicit) - no standard SOAP headers for transaction/security
  • No handlers other than SOAP Message Handlers

Test Approach

The test approach will utilize the JAX-RPC 1.0 Technology Release  to develop the JAX-RPC tests for both the JAX-RPC TCK and the J2EE 1.4 TS. The JAX-RPC tests requires a Servlet 2.3 container for execution. As stated above in the section Compatibility Test Suite Releases we will use the Apache TomCat 4.0 Web Server Platform environment for the JAX-RPC TCK and the J2EE 1.4 Platform environment for the J2EE 1.4 TS. Both platforms support the Servlet 2.3 specification.

Deliverables

  • Compatibility Test Plan for JAX-RPC
  • Suite of tests for both TCK and TS for JAX-RPC 1.0 functionality
  • Updated User's Guide for setup/config details of JAX-RPC tests.

Dependencies

The following dependencies are necessary for development of these tests:

  • JAX-RPC Reference Implementation (JAX-RPC RI)
  • JAX-RPC Version 1.0 Specification

Planning Assumptions

Notes to schedule above. 

  TCK focus is as follows and in this order:

  1) Signature Tests
  2) API Tests
  3) Specification and End-to-End Tests

  TCK will try to track RI team regarding functionality for JAXRPC. This
  however will not hold up TCK test development for API testing. Code will
  be written for published API's according to the Javadoc and will be tested
  as implemented in the specification.

  Rework of existing tests is in order due to specification changes from
  version to version regarding functional or api changes.

Risks

Risks to schedule above.

If there are late changing JAX-RPC specification changes or API changes
then this will slip the schedule by the amount of time to rework those tests
using these APIs.
